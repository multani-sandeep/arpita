**GCP — Conceptual Note for This Role**

*What GCP is and why it matters here*

- GCP (Google Cloud Platform) is one of the three major public cloud providers alongside AWS (Amazon) and Azure (Microsoft) — all offer the same core capabilities; the difference is vendor ecosystem and tooling names
- **Data warehouse:** GCP uses BigQuery — equivalent to Azure Synapse / SQL Data Warehouse (which Arpita named) or AWS Redshift
- **ML model platform:** GCP uses Vertex AI — equivalent to Azure Machine Learning or AWS SageMaker; where the photo compliance model is likely trained, versioned, and served
- **Data pipelines / ETL:** GCP uses Dataflow — equivalent to Azure Data Factory or AWS Glue; how raw photo data is processed and fed into the model
- **Object storage (photos):** GCP uses Cloud Storage — equivalent to Azure Blob Storage or AWS S3; where delivery photos are likely stored at scale
- **Streaming / real-time events:** GCP uses Pub/Sub — equivalent to Azure Event Hubs or AWS Kinesis; relevant if photo compliance results are pushed to couriers in real time
- **BI and reporting:** GCP uses Looker — equivalent to Power BI (Azure) or QuickSight (AWS); where compliance dashboards may be built
- **Monitoring:** GCP uses Cloud Monitoring / Operations Suite — equivalent to Azure Monitor or AWS CloudWatch; how system health and model performance are tracked post-launch

*PM actions to be mindful of in this role*

- When the engineering team references GCP service names (BigQuery, Vertex AI, Cloud Storage), understand what each one does at a functional level — not to configure them, but to ask the right questions about SLAs, data retention, and retraining cycles
- Model retraining on Vertex AI has a release cadence — treat it as a dependency in the delivery plan the same way a vendor firmware release is a dependency
- Photos stored in Cloud Storage = personal data at scale — GDPR and data retention policies need to be confirmed with Legal as a hard gate before go-live
- Cloud compute and storage costs are real budget line items — track GCP spend as part of financial forecasting, especially as rollout scales to more depots
- Model performance monitoring (false positive/negative rates post-launch) runs through GCP tooling — agree with the engineering team which dashboards exist and what thresholds trigger an escalation to you
- Go-live planning should account for GCP service availability — include it in the go/no-go checklist alongside operational readiness
- Jonathan has GCP certification — when he asks about the cloud stack, he is testing whether the PM understands the environment well enough to represent engineering constraints accurately to the business; not whether the PM can build in it
